Fase 1 · Meses 1-3
Fundación — Lo que bloquea la expansión
Sin estas bases, cualquier expansión regional va a colapsar. Son los cambios no negociables que hay que hacer antes de sumar un solo país nuevo.
⚡ Crítico
🔒
BLOQUEANTE
HTTPS obligatorio ▾
Hoy las contraseñas de red de Cencosud viajan en texto plano por la red. Cualquier persona en la misma red puede capturarlas con Wireshark en segundos.
seguridadnginxcertificado
Qué hay que hacer
Generar certificado SSL (certificado corporativo de Cencosud IT o Let's Encrypt si tiene salida a internet)
Configurar NGINX para escuchar en puerto 443 y redirigir HTTP → HTTPS automáticamente
Actualizar todas las URLs internas del sistema de http:// a https://
Configuración NGINX mínima
server { listen 443 ssl; ssl_certificate /etc/ssl/telecorp.crt; ssl_certificate_key /etc/ssl/telecorp.key; # resto de la config existente } server { listen 80; return 301 https://$host$request_uri; }
🖥️
BLOQUEANTE
Servidor dedicado o cloud ▾
El sistema corre en la PC de escritorio de tasalina. Si esa PC se reinicia, se apaga o tiene un problema, el sistema cae para todos los países.
infraestructuraazureaws
Opciones recomendadas
Azure VM (recomendado para Cencosud): Cencosud ya tiene contratos con Microsoft. Pedir una VM Windows Server con SQL Server en el tenant corporativo.
Servidor físico en datacenter Cencosud: Si IT tiene datacenter propio, pedir un servidor dedicado con Windows Server 2022.
Mínimo recomendado: 4 vCPU, 8GB RAM, 100GB SSD, acceso por dominio interno.
Qué cambia en el código
Eliminar rutas hardcodeadas: C:\Users\tasalina\Desktop\... → variables de entorno
El .bat de mantenimiento se configura como Tarea Programada del sistema, no del usuario
📁
BLOQUEANTE
Control de versiones con Git ▾
El código vive en un desktop. No hay historial de cambios, no se puede trabajar en equipo sin pisarse, y si se borra el archivo no hay forma de recuperarlo.
gitazure devopsgithub
Qué hacer
Crear repositorio en Azure DevOps (disponible en el tenant Cencosud) o GitHub Enterprise
Subir app.py, index.html, login.html y archivos de configuración
Agregar archivo .gitignore para no subir contraseñas ni archivos de base de datos
Regla de trabajo: cada cambio va a una rama, se revisa, se aprueba, y recién ahí se sube a producción
.gitignore mínimo
config.py # credenciales DB *.pyc __pycache__/ .env logs/ *.log
⚙️
IMPORTANTE
Variables de entorno ▾
Las credenciales de la base de datos y la configuración están escritas directamente en el código. Eso es un riesgo de seguridad y hace imposible tener ambientes distintos (dev, test, producción).
seguridadconfigpython
Crear archivo .env
DB_SERVER=servidor.cencosud.corp DB_NAME=TeleCorp DB_USER=telecorp_app DB_PASS=contraseña_segura SECRET_KEY=clave_aleatoria_larga LDAP_SERVER=cencosud.corp APP_ENV=production
En app.py
from dotenv import load_dotenv import os load_dotenv() DB_SERVER = os.getenv('DB_SERVER') SECRET_KEY = os.getenv('SECRET_KEY')

IMPORTANTE
Backup automático de la base de datos ▾
Hoy no hay backup automatizado. Un error en la base o un problema en el servidor implica pérdida total de datos históricos de gestiones, líneas y auditoría.
sql serverbackupautomatización
Estrategia recomendada
Backup completo diario a las 02:00hs (SQL Server Agent Job)
Backup diferencial cada 6 horas
Retención: 30 días locales + 90 días en Azure Blob Storage
Alerta por email si el backup falla

Fase 2 · Meses 3-6
Multi-país — Arquitectura para 5 países
Con la Fase 1 resuelta, ahora hay que rediseñar la base de datos y los permisos para soportar Argentina, Chile, Brasil, Perú y Colombia de forma aislada y segura.
🌎 Core
🗄️
BLOQUEANTE
Columna País en todas las tablas ▾
Agregar una columna Pais a dbo.Lineas, dbo.Gestiones, dbo.Maestro y dbo.Precios. Es el cambio más importante de toda la Fase 2 — todo lo demás depende de esto.
sql servermigraciónschema
Script de migración
ALTER TABLE dbo.Lineas ADD Pais NVARCHAR(3) DEFAULT 'ARG'; ALTER TABLE dbo.Gestiones ADD Pais NVARCHAR(3) DEFAULT 'ARG'; ALTER TABLE dbo.Maestro ADD Pais NVARCHAR(3) DEFAULT 'ARG'; ALTER TABLE dbo.Precios ADD Pais NVARCHAR(3) DEFAULT 'ARG'; ALTER TABLE dbo.Usuarios ADD Pais NVARCHAR(3) NULL; -- NULL = acceso global -- Actualizar registros existentes UPDATE dbo.Lineas SET Pais = 'ARG'; UPDATE dbo.Gestiones SET Pais = 'ARG'; UPDATE dbo.Maestro SET Pais = 'ARG'; UPDATE dbo.Precios SET Pais = 'ARG';
Códigos de país
ARG — Argentina
CHL — Chile
BRA — Brasil
PER — Perú
COL — Colombia
👤
BLOQUEANTE
Permisos por país en los roles ▾
Un admin de Argentina no puede ver ni modificar datos de Chile. Cada usuario debe tener un país asignado, y todas las queries del backend deben filtrar por ese país automáticamente.
rolesseguridadflask
Nuevos roles
superadmin: ve todos los países, acceso total (solo 1-2 personas)
admin_pais: admin de un país específico (ej: admin_ARG)
consulta_pais: solo lectura del país asignado
En app.py — filtro automático por país
def get_pais_usuario(): pais = session.get('pais') return None if pais == 'GLOBAL' else pais # En cada query: pais = get_pais_usuario() if pais: where.append("L.Pais = ?") params.append(pais)

IMPORTANTE
Selector de país en la interfaz ▾
Para usuarios con acceso a múltiples países (superadmin, gerentes regionales), agregar un selector de país en el header que filtre toda la vista en tiempo real.
ui/uxfrontendfiltros
Comportamiento esperado
Si el usuario tiene un solo país asignado: no ve el selector
Si tiene acceso a múltiples países: ve un dropdown en el header con las banderas
Al cambiar el país: todos los módulos (Dashboard, Líneas, Gestiones) filtran automáticamente
La selección queda en la sesión del usuario
🇧🇷
IMPORTANTE
Multi-idioma (i18n) ▾
Para Brasil se necesita portugués. Para el resto de los países el español neutro actual es suficiente. Hay que externalizar todos los textos de la interfaz a archivos de traducción.
i18nfrontendpt-BR
Enfoque recomendado
Crear archivos JSON de traducciones: es.json (español) y pt.json (portugués)
El idioma se asigna automáticamente según el país del usuario (BRA → pt, resto → es)
En el HTML, reemplazar textos hardcodeados por claves: data-i18n="lineas.alta.titulo"
Ejemplo es.json / pt.json
// es.json { "nueva_gestion": "Nueva gestión", "lineas": "Líneas" } // pt.json { "nueva_gestion": "Nova gestão", "lineas": "Linhas" }

Operaciones — Visibilidad y confiabilidad
El sistema ya corre en múltiples países. Ahora hay que saber qué está pasando en tiempo real, detectar errores antes de que los usuarios los reporten, y garantizar disponibilidad.
📊 Madurez

CRÍTICO
Monitoreo y alertas ▾
Hoy si el sistema cae, la primera persona en enterarse es un usuario que intenta acceder y no puede. Para producción regional necesitás saberlo antes que nadie.
monitoreoalertasuptime
Herramientas recomendadas
UptimeRobot (gratis): ping cada 5 minutos, alerta por email o Teams si cae
Azure Monitor (ya incluido en Azure): métricas de CPU, memoria y errores HTTP
Sentry (gratis hasta 5k errores/mes): captura errores de Python en tiempo real con stack trace completo
Integrar Sentry en app.py
import sentry_sdk sentry_sdk.init(dsn="tu-dsn-de-sentry") # A partir de aquí todos los errores no manejados # se reportan automáticamente con contexto completo
🧪
CRÍTICO
Tests automatizados de los endpoints críticos ▾
Hoy cualquier cambio en app.py puede romper el login, el alta de líneas o el procesamiento de bajas sin que nadie lo note hasta que un usuario falla. Los tests previenen eso.
pytesttestingci/cd
Endpoints a testear primero
POST /api/login — credenciales correctas, incorrectas, usuario inactivo
GET /api/lineas — responde, filtra por país, requiere login
POST /api/pedidos — crea gestión, valida campos requeridos
GET /api/dashboard — devuelve KPIs correctos
Ejemplo con pytest
def test_login_exitoso(client): r = client.post('/api/login', json={'usuario': 'test', 'password': 'pass'}) assert r.status_code == 200 assert r.json['ok'] == True def test_lineas_requiere_login(client): r = client.get('/api/lineas') assert r.status_code == 401

IMPORTANTE
Pipeline de deploy automatizado (CI/CD) ▾
Hoy para subir un cambio hay que copiar archivos manualmente al servidor. Con CI/CD, cada cambio aprobado en Git se despliega solo, con tests previos y rollback automático si algo falla.
azure devopsgithub actionsdeploy
Flujo de deploy
Developer sube cambio a rama feature/
Se crea Pull Request — revisor lo aprueba
Al mergear a main: el pipeline corre los tests
Si los tests pasan: deploy automático al servidor
Si algo falla: rollback al commit anterior automáticamente

reportesexcelpdf

MEJORA
Alta disponibilidad (failover) ▾
Para un sistema que opera en 5 países con diferentes husos horarios, hay que garantizar disponibilidad 24/7 con un mecanismo de failover automático si el servidor principal falla.
haazuresql server
Estrategia
SQL Server Always On Availability Groups: base de datos con réplica automática
Azure Load Balancer si se tienen múltiples instancias de la app
RTO (Recovery Time Objective) objetivo: menos de 5 minutos de caída

Recursos humanos
Equipo mínimo para sostener el sistema regional
TeleCorp no puede seguir siendo responsabilidad de una sola persona. Para un sistema regional necesita un equipo mínimo con roles claros.
💻
ROL
Desarrollador Python/Flask ▾
Responsable del backend, la base de datos, los tests y el deploy. Puede ser el mismo desarrollador actual más soporte de IT Cencosud.
Responsabilidades
Mantenimiento de app.py y endpoints
Migraciones de base de datos
Resolución de bugs en producción
CI/CD y deploy
🖥️
ROL
Infraestructura / IT Cencosud ▾
Gestión del servidor, backup, HTTPS, red y dominio interno. En Cencosud ya existe este equipo — hay que involucrarlo formalmente en el proyecto.
Responsabilidades
Servidor dedicado en Azure o datacenter
Certificado SSL corporativo
Backup y política de retención
Monitoreo de red y disponibilidad
🌍
ROL
Referente por país (admin_pais) ▾
Un referente operativo en cada país que conoce el negocio local, carga el Maestro mensual, gestiona usuarios de su país y reporta incidencias al equipo central.
Responsabilidades
Carga mensual del Maestro RRHH de su país
Alta y baja de usuarios del sistema en su país
Primer nivel de soporte para usuarios del país
Validación de datos y reporte de inconsistencias
